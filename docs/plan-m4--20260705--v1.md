## 0. 里程碑重定义（重要，先看这条）

原 `plan--20260704--v1.md` 的里程碑表把 **M4 = GIS 能力**、**M6 = LangGraph 编排**。经 2026-07-05 grill，**作者明确调整**：

- **M4 现在 = LangGraph 编排**（作者亲述的 #1 面试卖点，"最重要的部分是 agent 流程编排"）。

## 2. M4 目标与非目标

### 2.1 目标（做到即 M4 完成）
把现有能力用 **LangGraph `StateGraph`** 编排成一个**意图路由型科研 Copilot 总控图**：用户一句话 → Router 识别意图 → 路由到对应分支 → 关键节点人机确认（可中断、可恢复）→ 产出。图内包含 **1 个意图路由 + 3 个循环（重搜 / CRAG 自主补充 / 人机重答）+ 1 个人机中断**，体现"带环的动态图"这一编排价值，而非确定性 workflow。

**Demo（一条命令跑闭环，另有 resume 演示持久化）**：
```bash
# 流畅演示（单命令内中断-input-续跑）
uv run georesearcher --config config.m4.yaml orchestrate "学校隔离对教育不平等的影响"

# 硬核演示（跨进程断点续跑）：第一次跑到中断退出，打印 thread_id
uv run georesearcher --config config.m4.yaml orchestrate "..." --pause
# 第二次（全新进程）从 SQLite checkpoint 恢复
uv run georesearcher --config config.m4.yaml orchestrate --resume <thread_id> --feedback "补充欧洲数据"
```

### 2.2 非目标（M4 明确不做，避免范围蔓延）
- ❌ 不实现 PLOT / GIS / WRITE 三个节点的**真实能力**（各是后续独立里程碑）——它们在图中作为**占位节点**存在，节点体返回"该能力尚未实现（属 M5/MCP/M8 里程碑）"。
- ❌ 不接任何 MCP（GIS 占位节点里不真连 gis-mcp/qgis_mcp）。
- ❌ 不做全局纯 ReAct / 纯 Plan-and-Execute（过度设计，违背"编排式而非自主 multi-agent"决策）。
- ❌ 不改 M1/M2/M3 已实现能力的内部逻辑——编排层**只薄壳复用**它们的公开入口。
- ❌ 不追求 SEARCH 分支真连 OpenAlex（M3 已知被 Cloudflare TLS 阻断）——SEARCH 节点走**本地 PDF 投喂降级路径**演示，联网真跑作为"网络恢复后可用"记录。

## 3. 核心设计：三层 Agent 范式（本 milestone 的灵魂，面试主打）

> 一个成熟 agent 系统不该只用一种范式。M4 **分层用对范式**，每层都能说清"为什么是它、为什么不用另一种"。这是"真的懂 agent"的证明——不是用了最多范式，而是知道每种该用在哪。

| 层 | 范式 | 落在哪 | 为什么选它 / 为什么不用别的 |
|---|---|---|---|
| **总控层** | **Router / Intent-routing** | 顶层 `router_node` | 科研意图**有限且清晰**（找文献/问答/画图/GIS/写作）。对有限意图集，Router 分类比让 LLM 每步 ReAct 试错**更可控、更省 token、更好测**。不需要 agent 临场发明流程，只需选对已设计好的流水线。 |
| **RAG 质量控制层** | **Reflection / Corrective RAG (CRAG)** | `rag_qa_node` 后的"检索质量门控 → 不足则联网补充 → 重答"环 | 生成前先**评估检索质量**，不足则纠正性补充。用 **rerank 分数这个客观信号**做反思触发器，而非 LLM 自评（避免 M2 讲过的自我偏好偏差）；反思轮次有上限防失控烧钱。⭐ **CRAG 是 2024 热点范式，是本 milestone 最该主打的点。** |
| **工具编排层** | **ReAct（预留）** | 未来 `gis_node`（当前占位） | GIS 分析**步骤不确定**（看数据 schema→选算法→跑→看结果→可能换算法），才真正适合 ReAct（LLM 边看 MCP 返回边决定下一步）。把 ReAct **限定在 GIS 节点内部**，不污染总控——范式按任务特性分层。 |

**为什么不用全局纯 ReAct / 纯 Plan-and-Execute**：对"意图明确的科研流水线"是过度设计；自由 ReAct 不可控、成本翻倍、难测，违背 codebuddy.md 的"编排式而非自主 multi-agent"架构决策（决策 #2）。**范式是工具，按任务特性选。**

## 4. 图结构（要实现的最终图）

```
[*] → ROUTER(意图+置信度) ─┬─ 置信度<阈值 → CLARIFY(反问用户) ──→ 回 ROUTER
                          │
                          ├─ intent=SEARCH → SEARCH → (够 N 篇?)─否→ REFLECT(改写query) ⟲回SEARCH  【循环①重搜】
                          │                    │是
                          │                    ▼
                          │                 INGEST → INTERPRET → RAG_QA
                          │
                          ├─ intent=ASK ─────────────────────────→ RAG_QA
                          │                                          │
                          │                              retrieval_quality < 阈值 or 命中<N ?
                          │                              ├─是→ SEARCH(联网补充) ⟲(≤ crag_max_rounds) 【循环②CRAG自主补充】
                          │                              └─否→ 生成答案
                          │                                          │
                          ├─ intent=PLOT  → [占位:返回"M5实现"] ──┐    │
                          ├─ intent=GIS   → [占位:返回"MCP里程碑"]┼──→ HUMAN_REVIEW
                          └─ intent=WRITE → [占位:返回"M8实现"] ──┘         │
                                                             满意→[*] / 不满意带反馈→回 RAG_QA ⟲ 【循环③人机重答】
```

**三大编排特性对应**（面试指着图说）：
- **1 个动态路由**：ROUTER（+CLARIFY 澄清）——走哪条分支由 LLM 意图判断决定，不是写死 if-else。
- **3 个循环（带环动态图，DAG 函数链做不到）**：①检索不足自动改写重搜 ②CRAG 检索质量不足自主联网补充 ③人机不满意带反馈重答。
- **1 个人机中断**：HUMAN_REVIEW，`interrupt()` 暂停 + checkpoint 持久化 + 可跨进程 resume。

**防死循环**：每个循环有 round 计数字段（`search_round`/`crag_round`/`review_round`）+ config 上限，超限强制走 END 或 HUMAN_REVIEW。

```
src/georesearcher/orchestration/     ← 目录已存在（当前空 __init__.py）
├── __init__.py        # 导出 build_graph / ResearchState
├── state.py           # ResearchState（TypedDict + reducer）
├── router.py          # 意图识别：LLM 结构化输出 {intent, confidence, reason} + 解析
├── nodes.py           # 所有节点函数（薄壳复用能力层入口 + 占位节点）
└── graph.py           # 建图 + 条件边 + checkpointer + 编译
```

**分层红线**：orchestration 层**只依赖能力层的公开入口**（`ingest_from_search` / `interpret_paper` / `RetrievalPipeline` / `Generator`）和 store 接口，**不碰 HTTP、不直接 new 向量库、不写模型调用细节**。节点是薄壳：接收 state → 调能力层 → 把结果写回 state 的 key。能力层完全不知道 orchestration 存在（单向依赖）。

## 6. 关键接口签名（照抄，不要自创字段）

### 6.1 `orchestration/state.py` — 共享状态

```python
from __future__ import annotations
import operator
from typing import Annotated, TypedDict


class ResearchState(TypedDict, total=False):
    # ---- 输入 ----
    query: str                       # 用户原始输入
    config_path: str | None          # 透传给能力层的配置文件路径（--config）

    # ---- Router 层（Intent-routing 范式） ----
    intent: str | None               # "SEARCH" | "ASK" | "PLOT" | "GIS" | "WRITE"
    intent_confidence: float         # Router 置信度，< 阈值走 CLARIFY
    intent_reason: str               # Router 给出的判断理由（可观测/调试）
    clarify_round: int               # CLARIFY 反问轮次（防死循环）

    # ---- SEARCH / REFLECT 分支（循环①重搜） ----
    search_hits: list[dict]          # 轻量命中：[{paper_id, title, has_pdf}]
    search_round: int                # 重搜轮次
    refined_query: str | None        # REFLECT 改写后的检索词

    # ---- INGEST / INTERPRET ----
    ingested_ids: Annotated[list[str], operator.add]   # reducer 累加
    notes: Annotated[list[dict], operator.add]         # 结构化笔记（dict 化的 StructuredNote）

    # ---- RAG_QA + CRAG（Reflection 范式，循环②） ----
    rag_answer: str | None
    citations: list[dict]            # [{index, paper_id, reference}]
    retrieval_quality: float         # CRAG 门控信号 = 检索 top1 的 rerank 分数
    crag_round: int                  # CRAG 自主补充轮次

    # ---- HUMAN_REVIEW（中断，循环③） ----
    human_feedback: str | None       # 用户反馈；"" 或 None = 满意
    review_round: int                # 人机重答轮次

    # ---- 通用 ----
    error: str | None                # 节点失败降级信息（失败不崩，写进 state）
    trace: Annotated[list[str], operator.add]  # 节点执行轨迹（可观测/演示用）
```

> **reducer 说明**：`ingested_ids` / `notes` / `trace` 用 `Annotated[list, operator.add]`，多节点写入时**合并追加**而非覆盖；其余字段各节点各写各的 key（后写覆盖）。这是 LangGraph 状态合并的标准姿势——面试点"状态怎么传不污染"的答案。

### 6.2 `orchestration/router.py` — 意图识别（决策 C）

```python
from georesearcher.config import Config
from georesearcher.models.llm import LLMClient

VALID_INTENTS = ("SEARCH", "ASK", "PLOT", "GIS", "WRITE")

def classify_intent(query: str, *, llm: LLMClient, prev_clarify: str | None = None) -> dict:
    """LLM 结构化意图识别。
    返回 {"intent": str, "confidence": float, "reason": str}。
    - prompt 用 few-shot 锚定 5 类意图边界（见 _ROUTER_PROMPT）
    - 强制 JSON 输出；解析失败 → {"intent": "ASK", "confidence": 0.0, ...}（兜底走 CLARIFY）
    - prev_clarify：CLARIFY 反问后用户的补充，拼进 prompt 提高判准
    """

def _parse_intent_json(raw: str) -> dict:
    """剥 ```json 围栏 → json.loads → 校验 intent 在 VALID_INTENTS、confidence∈[0,1]；
    非法字段兜底为 intent=ASK, confidence=0.0（触发 CLARIFY）。纯函数，单测无网络。"""

_ROUTER_PROMPT: str  # 含 5 类意图定义 + 3~5 个 few-shot 例子；要求只输出 JSON
```

### 6.3 `orchestration/nodes.py` — 节点函数（薄壳，照抄能力层真实签名）

> 每个节点签名统一为 `def xxx_node(state: ResearchState, *, deps: NodeDeps) -> dict`，返回**要写回 state 的部分 key**（LangGraph 节点约定：返回 dict 会 merge 进 state）。`deps` 是依赖容器（见 6.4），便于测试注入 fake。

```python
# ---- Router 层 ----
def router_node(state, *, deps) -> dict:
    """调 classify_intent → 写 intent/intent_confidence/intent_reason。"""

def clarify_node(state, *, deps) -> dict:
    """置信度低时 interrupt 反问用户"你是想查已有知识库(ASK)还是找新文献(SEARCH)?"，
    拿到补充后 clarify_round+1，回 router 重判。"""

# ---- SEARCH / REFLECT 分支（复用 M3：ingest_from_search） ----
def search_node(state, *, deps) -> dict:
    """薄壳调 capabilities.search.pipeline.ingest_from_search(
           query=refined_query or query, limit=..., cfg=deps.cfg,
           sqlite_store=deps.sqlite_store, vector_store=deps.vector_store,
           embedder=deps.embedder, source=deps.search_source,   # 可注入本地降级源
           downloader=deps.downloader)
       → 把 IngestSearchResult 摊平进 search_hits / ingested_ids；失败写 error 不崩。"""

def reflect_node(state, *, deps) -> dict:
    """检索命中不足时，LLM 改写检索词 → 写 refined_query、search_round+1。"""

# ---- INGEST / INTERPRET（复用 M3：interpret_paper） ----
def interpret_node(state, *, deps) -> dict:
    """对 ingested_ids 逐个调 capabilities.interpret.interpret.interpret_paper(
           paper_id, cfg=deps.cfg, llm=deps.llm, sqlite_store=deps.sqlite_store)
       → StructuredNote.model_dump() 追加进 notes。"""

# ---- RAG_QA + CRAG（复用 M1：RetrievalPipeline + Generator） ----
def rag_qa_node(state, *, deps) -> dict:
    """1. results = deps.retrieval_pipeline.retrieve(query, config=deps.retrieval_cfg)
       2. retrieval_quality = results[0].child_chunk.score if results else 0.0   # rerank 分做 CRAG 门控
       3. 若 retrieval_quality >= 阈值 或 crag_round >= 上限:
            answer = deps.generator.generate(query, results)
            → 写 rag_answer / citations(由 answer.references 组装)
          否则: 只写 retrieval_quality（条件边会路由去 SEARCH 联网补充）"""

# ---- 占位节点（诚实边界，不实现能力） ----
def plot_node(state, *, deps) -> dict:
    return {"rag_answer": "[占位] 绘图能力属 M5 里程碑，编排接口已预留。",
            "trace": ["plot_node:placeholder"]}
def gis_node(state, *, deps) -> dict:
    return {"rag_answer": "[占位] GIS/QGIS 能力属后续 MCP 里程碑（预留 ReAct 子编排）。",
            "trace": ["gis_node:placeholder"]}
def write_node(state, *, deps) -> dict:
    return {"rag_answer": "[占位] 学术写作能力属 M8 里程碑，可接外部 skill。",
            "trace": ["write_node:placeholder"]}

# ---- 人机中断（决策 B / b+） ----
def human_review_node(state, *, deps) -> dict:
    """feedback = interrupt({"answer": state["rag_answer"],
                             "citations": state.get("citations", []),
                             "prompt": "满意吗？回车通过，或输入追问让我重答"})
       - feedback 为空/None → 写 human_feedback=None（条件边走 END）
       - 否则 → 写 human_feedback=feedback、review_round+1（条件边回 RAG_QA）"""
```

### 6.5 `orchestration/graph.py` — 建图与编译

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver          # 单命令内 resume
# from langgraph.checkpoint.sqlite import SqliteSaver        # 跨进程持久化（b+）
from georesearcher.orchestration.state import ResearchState

def build_graph(deps, *, checkpointer=None):
    """组装 StateGraph：
    - add_node 全部节点（用 functools.partial 把 deps 绑进节点函数）
    - add_edge / add_conditional_edges 按 §4 图结构连边
      · router 后：按 intent + confidence 路由（含 CLARIFY 分支）
      · search 后：够 N 篇 → interpret；否则 → reflect（search_round 上限保护）
      · rag_qa 后：retrieval_quality 门控 + crag_round 上限 → search补充 或 human_review
      · human_review 后：human_feedback 空 → END；否则 → rag_qa（review_round 上限保护）
    - checkpointer = checkpointer or MemorySaver()
    - return workflow.compile(checkpointer=checkpointer)
    """

# 条件边路由函数（纯函数，单测重点）：
def _route_after_router(state: ResearchState) -> str: ...   # → "clarify"|"search"|"rag_qa"|"plot"|"gis"|"write"
def _route_after_search(state: ResearchState) -> str: ...   # → "reflect"|"interpret"
def _route_after_rag(state: ResearchState) -> str: ...      # → "search"|"human_review"
def _route_after_review(state: ResearchState) -> str: ...   # → "rag_qa"|END
```

### 6.6 CLI 新增命令（`ui/cli.py`，与现有 `ask`/`ingest-search` 同风格）

```python
@app.command()
def orchestrate(
    query: str = typer.Argument(None, help="研究主题或问题；--resume 时可省略"),
    resume: str = typer.Option(None, help="从 checkpoint 恢复的 thread_id（跨进程）"),
    feedback: str = typer.Option(None, help="--resume 时注入的人机反馈"),
    pause: bool = typer.Option(False, help="跑到人机中断即退出并打印 thread_id（演示持久化）"),
    config: str = typer.Option(None, help="配置文件路径，默认 config.yaml"),
):
    """运行 Agent 编排总控图。
    - 无 resume：新会话，stream 到 human_review 中断
        · 默认（无 --pause）：单命令内捕获中断 → input() 拿反馈 → Command(resume=...) 续跑（流畅 demo）
        · --pause：打印 answer + thread_id 后退出（状态存 SqliteSaver，可跨进程恢复）
    - 有 resume：全新进程，用 SqliteSaver 按 thread_id 恢复 → Command(resume=feedback) 续跑
    rich 打印：意图路由结果、trace 轨迹、最终答案 + 引用。"""
```

## 10. ADR 追加（写进 design §11）

**ADR-16 Agent 范式分层选型**：选 **三层范式**——总控层 = Router/Intent-routing；RAG 质量控制层 = Reflection / Corrective RAG（rerank 分数客观信号触发，非 LLM 自评）；工具编排层（GIS）= 预留 ReAct。备选：全局纯 ReAct、全局纯 Plan-and-Execute。理由：科研意图有限且清晰，Router 比 ReAct 更可控/省 token/可测；质量控制用客观信号避免自我偏好偏差（呼应 M2）；开放式工具编排才用 ReAct，限定在节点内部不污染总控。Trade-off：不如纯 ReAct"自主"，但可控、可测、省钱，符合"编排式而非自主 multi-agent"（决策 #2）。

**ADR-17 编排骨架完整 + 能力按里程碑填（占位节点）**：M4 实现完整 5 分支意图路由骨架，但只真接 SEARCH+ASK（M1/M3 能力），PLOT/GIS/WRITE 为占位节点（返回"未实现"）。备选：只画图不接 / 5 条全实现。理由：骨架完整体现设计能力，占位体现编排/能力解耦（填能力不动编排层）+ 诚实边界。Trade-off：demo 有 3 个分支是占位，但这是可讲的加分项而非硬伤。

**ADR-18 CRAG 用检索分数门控自主补充**：RAG_QA 是否触发联网补充，由 `retrieval_quality = top1 rerank 分 < crag_quality_threshold`（config）决定，非 LLM 自评；`crag_max_rounds` 上限防失控。备选：让 LLM 判"够不够答"。理由：客观信号避免 LLM 自我偏好偏差，可量化、可复现（呼应 M2 的 LLM-as-judge 三坑）。Trade-off：阈值需调参，但比 LLM 拍脑袋可控。

**ADR-19 人机中断用 interrupt + SqliteSaver 持久化（b+）**：HUMAN_REVIEW 用 LangGraph 标准 `interrupt()`，checkpointer 用 SqliteSaver 持久化，支持单命令内 resume（流畅）与跨进程 `--resume`（持久）。备选：节点内 `input()`（无 LangGraph 特性）、仅 MemorySaver（不能跨进程）。理由：既流畅演示又能讲"断点续跑、状态持久化 SQLite"。Trade-off：多一个 checkpoint-sqlite 依赖 + 测试量最大。

## 11. 红线与禁止（M4 特化）

- ❌ 不实现 PLOT/GIS/WRITE 的真实能力（占位节点只返回预设串）。
- ❌ 不接任何 MCP（GIS 占位节点不真连 gis-mcp/qgis_mcp）。
- ❌ 不做全局纯 ReAct / 纯 Plan-Execute（违背编排式决策）。
- ❌ orchestration 层不碰 HTTP、不直接 new 具体向量库/模型；只走能力层入口 + store 接口（分层红线）。
- ❌ 不改 M1/M2/M3 能力的内部实现（编排只薄壳复用）。
- ❌ 单测不发真实网络、不下模型、不真调 DeepSeek（全 mock）。
- ❌ CRAG 补充/人机重答/重搜/澄清**必须有 round 上限**，禁止无上限循环（防烧钱/死循环）。
- ❌ 除 `langgraph` + `langgraph-checkpoint-sqlite` 外，不引入其他新重依赖（先问）。

## 12. 交付后可讲的面试话术（洗清楚，可直接背，供 interview 文档取材）

### 12.1 一句话定位
"M4 我用 LangGraph 把前面做的 RAG、评估、检索入库这些孤立能力，编排成了一个**意图路由型的科研 Copilot 总控图**——用户一句话，agent 识别意图路由到不同分支，能自主判断检索质量决定要不要联网补充，关键节点还能人机确认、断点续跑。"

### 12.2 核心追问：这跟 if-else + 顺序调函数有啥区别？（必被问）
"区别在**带环的动态图**，不是确定性 workflow。三个地方普通函数链做不到：① Router 让走向由 LLM 意图判断决定，不是写死；② 有循环——检索不足自动改写重搜、检索质量不够自主联网补充、答案不满意带反馈重答，这是 DAG 函数链表达不了的；③ 用 `interrupt` + checkpointer 做人机中断和断点续跑，状态持久化在 SQLite，能跨进程恢复。这些是 LangGraph 图模型独有的，函数链给不了。"

### 12.3 核心追问：你这是什么 agent 范式？（拉开差距的点）
"我没教条套单一范式，而是**分层用对范式**：
- **总控层用 Router / Intent-routing**——科研意图有限且清晰，Router 比 ReAct 更可控、更省 token、更好测，我不需要 agent 临场发明流程，只需选对已设计好的流水线；
- **RAG 质量控制层用 Reflection，具体是 Corrective RAG(CRAG)**——生成前先用 **rerank 分数这个客观信号**判检索覆盖度，不足才纠正性地联网补充，而不是让 LLM 自评'够不够'（避免自我偏好偏差，这是我 M2 评估里踩过的坑），反思有轮次上限防失控；
- **开放式工具编排（GIS）预留 ReAct**——空间分析步骤不确定，才需要边看工具返回边决定下一步，我把 ReAct 限定在 GIS 节点内部不污染总控。
我**刻意没让整个系统跑纯 ReAct 或纯 Plan-Execute**——那对意图明确的科研流水线是过度设计，也违背我'编排式而非自主 multi-agent'的架构决策（自主 ReAct 不可控、成本翻倍、难测）。**范式是工具，按任务特性选，不是越自主越高级。**"

### 12.4 追问：Router 判错意图怎么办？（头号软肋，必被问）
"我做了三层防御：① Router 输出**结构化意图 + 置信度**，低于阈值（config 默认 0.6）就走 CLARIFY 节点反问用户'你是想查已有知识库还是找新文献'；② 意图是 state 里的字段，人机确认时用户能看到'我判成了 SEARCH'，不对可以纠正；③ Router 的 prompt 用 few-shot 例子锚定 5 类意图边界。解析失败也兜底成低置信度走澄清，不会硬判。"

### 12.5 追问：'知识库不够答就联网'——'不够'谁判的？（第二软肋，必被问）
"不靠 LLM 拍脑袋说'够不够'，用**检索层的客观信号门控**：检索 top1 的 rerank 分数低于阈值（config 默认 0.3）就判定覆盖不足，触发联网补充。这复用了我 M2 评估的检索质量思路——**用可量化信号而非 LLM 自评做决策**。而且设了最大补充轮次（默认 1），防止无限搜索烧钱失控。"

### 12.6 追问：占位节点算做完了吗？（会被问）
"编排层和能力层解耦是我刻意的设计。绘图、GIS、写作各是独立里程碑，我在编排骨架里留了占位节点和统一接口，填能力时不动编排层——这正是'编排是编排、能力是能力'的分层价值。我宁愿骨架完整、能力按里程碑填，也不愿凑一个假的画图 demo。"

### 12.7 追问：状态怎么在节点间传？会互相污染吗？
"共享 `ResearchState`（TypedDict），累加类字段（ingested_ids/notes/trace）用 `Annotated[list, operator.add]` 的 reducer 合并，其余字段各节点各写各的 key。节点只读写自己关心的 key，不直接调别的节点——**通过 state 通信而非 agent 互相喊话**，这就是编排式（而非自主 multi-agent）的核心。"

### 12.8 追问：为什么用 LangGraph 不用 CrewAI / AutoGen？
"我要的是**编排式**（节点即专职 agent、state 传递、流程可控可观测），不是 CrewAI/AutoGen 那种角色自由对话的自主 multi-agent（失控 + 成本翻倍 + 难测）。LangGraph 的图模型天然表达条件边和循环，`interrupt`+checkpointer 原生支持人机协同和断点续跑——这些正好匹配我'确定性可控 + 可观测 + co-pilot'的诉求。"

## 0. M2 是什么、为什么重要

**目标**：给 M1 的 RAG 内核配一个**可复现的分层评估闭环** —— 输入一份评估集，输出一份"检索层 / 生成层"分层指标报告，并能指出**"是检索层弱还是生成层弱"**，支持改动前后回归对比。

**为什么是最高价值里程碑**：作者面试被追问"**如何评测 RAG 效果**"（见 codebuddy.md §3）。M2 做完，这个问题就能拿**实物 demo + 话术**回答。这是简历/面试的重头戏（design §5.2、plan 里程碑表 M2 标 ⭐ 最高价值）。

**Demo 目标**：
```bash
georesearcher eval                       # 跑默认评估集
# → 终端打印分层指标表格（rich）
# → 生成 data/reports/eval--<时间戳>.md（人可读报告 + 定位建议）
# → 生成 data/reports/eval--<时间戳>.json（机器可读，供回归对比）

georesearcher eval-diff <run_a.json> <run_b.json>   # 回归对比两次结果
# → 打印指标 diff（哪些涨了、哪些跌了、防跷跷板）
```

| # | 决策点 | 结论 |
|---|--------|------|
| D1 | 评估集来源 | **LLM 半自动生成 + 人工校验**。judge LLM 从已入库论文造 QA 对并标注来源，再由作者人工删改。 |

### 阶段 A：前置修复（M1 缺陷，评估强依赖）

#### ☐ T0 — 给 `RetrievalPipeline` 加 `retrieve_raw()` + HyDE 容错

**为什么**：评估检索层指标（hit@k/MRR/NDCG）需要**未经父块扩展的原始 `list[Retrieved]`**，才能拿到每个命中的 `chunk.paper_id` 和排名。现在 `retrieve()` 只返回 `RetrievalResult`（已父块化），拿排名不方便。

**做什么**（改 `src/georesearcher/capabilities/rag/retriever.py`）：
1. 把 `retrieve()` 里"检索+融合+重排序得到 `final: list[Retrieved]`"的逻辑抽成一个**内部方法** `_retrieve_ranked(query, rc) -> list[Retrieved]`（第 328~403 行那段：tag 过滤 → 多查询 → 稠密+稀疏 → RRF → 重排序 → `final = candidates[:rc.top_k]`）。
2. 新增**公共方法**：
   ```python
   def retrieve_raw(self, query: str, *, config: RetrievalConfig | None = None,
                    skip_cache: bool = False) -> list[Retrieved]:
       """返回排序后的原始命中（父块扩展前），供评估用。"""
   ```
   它调用 `_retrieve_ranked` 并返回 `list[Retrieved]`（**不做 `_results_to_retrieval_results`**）。
3. `retrieve()` 保持原行为不变：内部改为 `raw = self._retrieve_ranked(...)`，再 `return self._results_to_retrieval_results(raw)`。**回归测试确保 `retrieve()` 输出不变。**
4. **HyDE 容错**：`_generate_hyde` 目前无 try/except（第 487~489 行）。包一层：
   ```python
   def _generate_hyde(self, query: str) -> str:
       try:
           return self._llm.complete(_HYDE_PROMPT.format(question=query), temperature=0.7)
       except Exception:
           return ""   # 失败则退回纯 query 检索，不让整条检索崩
   ```
   同样给 `_generate_multi_queries` 包容错（失败退回 `[query]`）。

**DoD**：`test_retrieve_raw.py`：① `retrieve_raw` 返回 `list[Retrieved]` 且长度 ≤ top_k；② mock `_llm.complete` 抛异常时 HyDE 不崩、能返回结果；③ `retrieve()` 行为与改前一致（可用 mock 各子步验证返回结构）。测试**不得**真调 DeepSeek（用 monkeypatch mock LLM/embedder/vector_store）。

#### ☐ T1 — 补 LLM judge 重排序（`_rerank`）的单元测试

**为什么**：`_rerank`（第 501~529 行，交叉编码器禁用时的 fallback）**从未被测过**，且输出解析靠正则，脆弱。M2 若切强模型做 judge，可能触发此路径。

**做什么**：新增测试（可并入 `test_retrieve_raw.py` 或单开 `test_rerank.py`）：
- mock `self._judge.complete` 返回典型格式 `"passage_0: 8\npassage_1: 3\n..."`，断言重排序后顺序正确。
- mock 返回**畸形/空**输出（如乱码、无 `passage_` 行），断言不崩、退回原顺序 `candidates[:top_k]`。

**DoD**：两个测试用例绿，覆盖正常 + 畸形两条路径。**只加测试，非必要不改 `_rerank` 逻辑**（若发现明确 bug 才最小修复并说明）。

> ⚠️ 红线：T0/T1 是评估必需的最小修复。**不要**顺手改 tokenizer、不要把 cross-encoder 改成可配置、不要重写 tag 提取 —— 那些留到以后，本次 M2 不做（决策 D4）。

### 阶段 B：评估集（M2 地基）

#### ☐ T2 — 定义评估数据结构 `schemas.py`

**做什么**（`capabilities/evaluation/schemas.py`，用 pydantic v2 BaseModel）：
```python
class EvalCase(BaseModel):
    id: str                              # 唯一 id，如 "q001"
    question: str
    expected_paper_ids: list[str]        # ground truth（paper 级，决策 D2）
    reference_answer: str | None = None  # 可选：生成层评估参考答案
    tags: list[str] = []                 # 可选：便于分层看指标

class RetrievalMetrics(BaseModel):
    hit_rate: float                      # 命中率（top_k 内是否有 GT paper）
    mrr: float                           # Mean Reciprocal Rank
    ndcg: float                          # NDCG@k
    context_precision: float             # top_k 中 GT paper 的占比（去重 paper 后）
    context_recall: float                # GT paper 被检回的比例
    k: int                               # 评估用的 top_k

class GenerationMetrics(BaseModel):
    faithfulness: float                  # 答案是否忠于检索上下文（0-1）
    answer_relevancy: float              # 答案是否切题（0-1）
    n_evaluated: int

class CaseResult(BaseModel):             # 单条 case 的明细（便于报告/debug）
    case_id: str
    question: str
    retrieved_paper_ids: list[str]       # 实际检回（去重、按排名）
    expected_paper_ids: list[str]
    hit: bool
    reciprocal_rank: float
    answer: str | None = None
    faithfulness: float | None = None
    answer_relevancy: float | None = None

class EvalReport(BaseModel):
    timestamp: str                       # ISO8601
    n_cases: int
    retrieval: RetrievalMetrics
    generation: GenerationMetrics | None # 未跑生成层时为 None
    ragas: dict | None = None            # 可选 RAGAS 对照
    config_snapshot: dict                # 本次跑的 retrieval 配置（可复现）
    cases: list[CaseResult]
    diagnosis: str                       # "检索层弱还是生成层弱"的文字定位
```
**DoD**：`test_eval_schemas.py`：各模型 `model_validate` / `model_dump` roundtrip 通过。

#### ☐ T6 — 造评估集脚本 `scripts/gen_eval_set.py`（+ 必要的 SQLite 方法）

> 编号 T6 但建议在 T3 后做（它产出的 JSON 喂给 T3 加载）。

**前置检查**：`grep -n "def " src/georesearcher/storage/sqlite_store.py`。造集脚本需要：
- **列出全部论文**（`list_papers() -> list[Paper]` 或返回 id+title）。若没有，**在 sqlite_store 补一个** `list_papers(limit=None) -> list[Paper]`（简单 `SELECT * FROM papers`），并加最小测试。
- **取某论文的正文**做造题素材：可用 `get_parent_chunk(paper_id, section_idx)` 循环，或补 `get_parent_chunks_for_paper(paper_id) -> dict[int,str]`（若已存在则直接用——报告显示 batch_ingest 用过类似方法，先 grep 确认）。

**脚本流程**（`scripts/gen_eval_set.py`，纯脚本，不进 package）：
1. 连 SQLite，抽取论文（可按 tag 分层抽样，保证主题覆盖：教育不平等/空间分析/方法 等）。
2. 对每篇（或每组）论文，取其 section 全文若干段，喂给 `get_judge()` LLM，用**造题 prompt**生成 1~2 个"该论文能回答的研究问题" + 简短参考答案。
3. 每条 QA 的 `expected_paper_ids` = 造题所用论文的 id（**这是 GT 的来源**）。
4. 输出 `tests/eval_set/eval_set.draft.json`（草稿，**不直接覆盖定稿**）。
5. 脚本打印提示："请人工校验草稿，删掉不合理的题、修正 expected_paper_ids，另存为 eval_set.json"。

**造题 prompt 要点**（写进脚本常量，英文或中文均可，与语料一致）：
- 要求生成"**具体、可被这篇论文回答**"的研究问题，避免泛泛而谈。
- 要求问题**不要**直接抄论文标题句式（否则检索太容易，指标虚高）。
- 输出结构化（JSON），便于解析；judge temperature=0.0。

**规模**：脚本默认造 ~40 条草稿（多造点，人工校验会删掉一些，最终留 ~30 条，决策 D5）。

**DoD**：
- 脚本能跑（有 DEEPSEEK_API_KEY 时），产出 `eval_set.draft.json` 结构合法（能被 T3 的 `load_eval_set` 加载）。
- **人工校验后**，作者定稿 `tests/eval_set/eval_set.json`（~30 条，入 git）。
- `tests/eval_set/README.md` 写清格式 + 重造/校验步骤。
- ⚠️ 执行者：**造集脚本你写，但最终 eval_set.json 的定稿需要作者人工校验后确认**。写完脚本、产出草稿后**停下来交给作者校验**，不要拿未校验的草稿当定稿去跑指标（决策 D1）。

### 阶段 C：评估指标

#### ☐ T4 — 检索层指标 `retrieval_metrics.py`（纯计算，不碰 LLM）

**做什么**：实现 paper 级指标（决策 D2）。输入：每条 case 的 `expected_paper_ids` + 检索实际返回的**去重 paper_id 排序列表**（从 `retrieve_raw()` 的 `list[Retrieved]` 提 `chunk.paper_id`，按排名去重）。

```python
def dedup_paper_ranking(retrieved: list[Retrieved]) -> list[str]:
    """按排名把 chunk 命中折叠成 paper_id 排序列表（首次出现的排名代表该 paper）。"""

def hit_rate(pred_papers: list[str], gold_papers: list[str], k: int) -> float
def mrr(pred_papers: list[str], gold_papers: list[str], k: int) -> float
def ndcg(pred_papers: list[str], gold_papers: list[str], k: int) -> float
def context_precision(pred_papers: list[str], gold_papers: list[str], k: int) -> float
def context_recall(pred_papers: list[str], gold_papers: list[str], k: int) -> float

def aggregate_retrieval(per_case: list[dict], k: int) -> RetrievalMetrics
    """对所有 case 的单条指标取平均，组装 RetrievalMetrics。"""
```
- **hit_rate**：top_k paper 里是否至少命中一个 gold → 1/0，再对全部 case 取均值。
- **mrr**：第一个命中 gold 的 paper 的排名倒数（1/rank），无命中记 0，取均值。
- **ndcg@k**：gold paper 命中记 relevance=1，按标准 DCG/IDCG 公式；paper 级、二元相关性。
- **context_precision**：top_k 去重 paper 中属于 gold 的比例。
- **context_recall**：gold paper 中被检回（在 top_k 内）的比例。

**DoD**：`test_retrieval_metrics.py`：**用手工构造的 pred/gold 列表**验证每个指标的数学正确性（如 gold 在第 2 位 → MRR=0.5；全命中 → recall=1.0；边界：无命中、gold 为空、k 大于列表长度）。**不碰 LLM、不碰真实检索**，纯数学单测。这是最容易测准、最能体现工程严谨的部分。

#### ☐ T5 — 生成层指标 `generation_metrics.py`（自研 LLM-as-judge）

**为什么自研**（决策 D3）：可控、能配 DeepSeek、能处理 LLM-as-judge 三个坑（design §5.2）。RAGAS 作为可选对照放 T7。

**做什么**：实现两个指标，都走 `get_judge()`：

1. **Faithfulness（忠实度 / groundedness）**：答案的每个断言是否被检索上下文支持。
   - 做法：给 judge "上下文 + 答案"，让它逐句判断"是否被上下文支持（是/否）"，忠实度 = 支持句数 / 总句数。
   - **用是/否细粒度问题替代 1-5 打分**（design §5.2 第 3 坑：评分不一致）。
2. **Answer Relevancy（答案相关性）**：答案是否切题（回答了 question）。
   - 做法：给 judge "question + 答案"，让它判断答案是否直接、完整回答了问题（是/部分/否 → 1/0.5/0），或用"根据答案反推可能的问题、与原问题相似度"的经典 RAGAS 思路（简化版即可）。

**三个坑的对策（写进代码 + 注释，面试要讲）**：
- 坑1 自我偏好偏差 → judge model 可配置（`config.models.judge`），验证期切强模型交叉；报告里标注用的 judge model。
- 坑2 位置/长度偏差 → 单条评估、控制上下文截断长度一致（复用 M1 的 1500 字符截断）。
- 坑3 评分不一致 → judge **temperature=0.0**（已是 judge 默认）、用**是/否二元判断**而非 1-5 主观分。

```python
class GenerationJudge:
    def __init__(self, judge: LLMClient | None = None): ...
    def faithfulness(self, answer: str, contexts: list[str]) -> float: ...
    def answer_relevancy(self, question: str, answer: str) -> float: ...

def aggregate_generation(per_case: list[dict]) -> GenerationMetrics: ...
```
- judge 输出**解析要健壮**（学 T0/T1 教训）：解析失败时该条记为缺失、不计入均值，别让整轮崩。

**DoD**：`test_generation_metrics.py`：**mock `judge.complete`** 返回固定文本，断言解析与聚合正确（如 3 句里 2 句"是" → faithfulness≈0.67）；解析畸形输出时不崩。**不真调 DeepSeek。**

### 阶段 D：编排、报告、CLI

#### ☐ T8 — 评估编排器 `evaluator.py`

**做什么**：串起来。
```python
class RAGEvaluator:
    def __init__(self, cfg=None, pipeline: RetrievalPipeline | None = None,
                 generator: Generator | None = None, judge: LLMClient | None = None): ...

    def run(self, cases: list[EvalCase], *, top_k: int = 5,
            eval_generation: bool = True, run_ragas: bool = False,
            retrieval_config: RetrievalConfig | None = None) -> EvalReport:
        """对每条 case：retrieve_raw → 算检索指标；（可选）generate → 算生成指标。
           聚合 → 生成 diagnosis → 返回 EvalReport。"""
```
流程：
1. 对每条 case：`raw = pipeline.retrieve_raw(case.question, config=retrieval_config)` → `dedup_paper_ranking` → 存单条检索结果。
2. 若 `eval_generation`：`results = pipeline.retrieve(case.question)`（要父块）→ `answer = generator.generate(...)` → `GenerationJudge` 算 faithfulness/relevancy。
3. 聚合 `aggregate_retrieval` + `aggregate_generation`。
4. **diagnosis 逻辑**（面试锚点，写清阈值）：比较检索层与生成层的相对强弱，给文字建议，例如：
   - 检索 hit_rate 低（<0.6）→ "检索层弱：建议调分块/embedding/混合检索权重/HyDE/多查询"。
   - 检索好但 faithfulness 低 → "生成层弱：建议调 prompt/强制引用/缩上下文/降温"。
   - 阈值写成模块常量、可在 config 覆盖，注释说明这些是经验阈值、非绝对。
5. `config_snapshot` 存本次 retrieval 配置，保证可复现。

**DoD**：`test_evaluator.py`（可选但推荐）：用 mock pipeline/generator/judge 跑 2~3 条构造 case，断言 `EvalReport` 结构完整、diagnosis 非空。**不真调网络。**

## 5. 执行顺序与里程碑内 checkpoint

**依赖顺序**（严格照此，前一个绿了再下一个）：

```
T0 (retrieve_raw + HyDE容错)  ──┐
T1 (rerank 测试)               ─┤  阶段A：前置修复
                                │
T2 (schemas) → T3 (dataset) ────┤  阶段B：地基（T2/T3 可先于造集）
T6 (gen_eval_set 脚本) → 【人工校验 eval_set.json】
                                │
T4 (检索指标, 纯数学)  ─────────┤  阶段C：指标（T4 不依赖 LLM，最先可测）
T5 (生成指标, mock judge)      ─┤
T7 (RAGAS 可选)                 │
                                │
T8 (evaluator) → T9 (report) → T10 (CLI)   阶段D：串联
                                │
T11 (config) ── 与 T8 并行可做  ┤  阶段E
T12 (依赖) / T13 (文档)         ─┘
```

**建议先跑通的最小闭环**（给弱模型一个早期成就点 & 早暴露问题）：
> T0 → T2 → T3 → 用 **3 条手写 case**（不等 T6 造集）→ T4（检索指标）→ 一个简版 `evaluator.run(eval_generation=False)` → 终端打印检索指标。
> 这条"只检索层、纯本地、不调生成 LLM"的链路能最快验证 pipeline 通不通，再逐步加生成层、报告、CLI、真评估集。
